# 通用执行规范（SOP）

> 适用于所有项目、所有任务。违反视为无效。

## 步骤

### 0. 前置分类：判断任务类型

收到请求后，先判断属于哪一类：

| 类型 | 特征 | 处理方式 |
|------|------|---------|
| **咨询型** | 查信息、问概念、求建议 | 直接回答，无需方案 |
| **设计型** | 建新机制、定新规范、做方案选型 | 先出方案沟通确认，再动手 |
| **执行型** | 修 bug、改文案、按已知流程操作 | 直接执行，无需方案 |

> 拿不准的按**设计型**处理。

### 1. 设计型任务 → 先方案后执行

1. **出方案**：输出 1~3 段方案描述（做什么、怎么做、备选）
2. **沟通确认**：等你确认再进入执行
3. **确认后补三文档**：方案确认后写入 `docs/`（如适用）
4. **转执行型**：按后续步骤执行

### 2. 执行型任务 → 按原流程

1. **创建工作目录**：项目根目录创建 `_task/`，所有生成文件放此处（在家目录 `~` 时跳过）
2. **前置读取**：顺序读 `docs/PRD.md` → `docs/DESIGN.md` → `docs/TASKS.md`
   - 不存则沟通 → 我生成初始内容 → 你确认
3. **执行任务**（`_task/` 内操作）
4. **修改三文档前先备份**：执行 `cleanup --backup-only`
5. **完成后同步更新**：TASKS.md 状态 + DESIGN.md 技术方案
6. **收尾清理**：`cleanup` 删 `_task/`、清旧备份、删垃圾文件
7. **废弃清理**：方案跑不通换方向时，`cleanup`

## 禁止行为

- 未读三文档直接编码
- 推荐或实现 DESIGN.md 中"已否决方案"
- 静默重构不记录原因
- 不明确需求自行假设

## 审核职责

- **设计决策/技术选型**：主动审查疏漏，指出与 DESIGN.md 矛盾或已否决的方案
- **模糊表述**：必须反问澄清
- **做完了报告**：做了什么、更新了什么、是否 cleanup、遗留风险

## 验证

- 每次开始前确认三文档已读
- 完成后确认 TASKS.md 已更新
- 方案变更确认 DESIGN.md 同步更新
